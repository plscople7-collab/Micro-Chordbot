【概要】

* ガバく見える主因は、機能量に対して「階層・主従・操作導線」が整理されていないこと。
* 画面上に、編集対象・ライブラリ・再生・設定・入出力・デバッグ情報が同じ重さで並んでいる。
* 音高編集とコード進行編集でUI思想が統一されておらず、同じアプリ内で別物感が出ている。
* ボタン・入力欄・リスト・タグ・カードの視覚ルールが揃っていないため、どこを触れば何が変わるかが直感的に掴みにくい。
* 現状は「開発者用操作盤」としては使えるが、「作曲支援ツール」としては情報密度が制御されていない。

【詳細解説】
まず大きいのは、画面の責務分離が弱い点です。1枚目では上部に鍵盤、音高ライブラリ、音高プリセット、コードライブラリが縦に並んでいますが、それぞれが「現在操作中の音」「保存済み音高」「追加候補」「コード定義」という別階層の情報です。本来なら、現在値・候補・保存済み・編集履歴は視覚的に分けるべきですが、今はすべて同じパネル調で並ぶため、ユーザーから見ると「どれが現在の作業対象なのか」が曖昧になります。特に activeNotes と音高プリセットの境界が弱く、上は編集対象、下は候補一覧であるにもかかわらず、表形式・タグ・メモ欄が似た密度で表示されているため、操作意味の差が見えにくいです。
2枚目のコード進行画面は、さらに「編集領域」と「再生・保存・読込・プレビュー領域」の分離が中途半端です。左側はルート、拍、Bass、転回形、構成音ボタン、コード種ボタンが一塊になっていますが、ユーザーの思考順はおそらく「コードを選ぶ→拍を決める→配置を決める→進行へ追加する」です。しかしUI上では、Section、ルート、拍、臨時記号、Bass、転回形、度数選択、ルートボタン、コード品質ボタンが一つの巨大な編集盤面に押し込まれています。結果として、コードを作る操作が「音楽的選択」ではなく「パラメータ群の手入力」に見える。これはμChordbotの思想自体が悪いのではなく、内部モデルの豊かさがそのまま露出している状態です。
また、視覚的な主従関係が弱いです。現在選択中のコード、現在のチャンク、再生位置、編集対象、保存対象がすべて小さな色差で表現されています。たとえば右側のコード進行カードでは、選択中の「A m」は枠色で示されていますが、背景や周囲カードとの差が弱く、パッと見で「今ここを編集している」という確信が持ちにくいです。音高画面でも、鍵盤上の点、activeNotesの行、下部プリセットのハイライトがそれぞれ別のルールで表示されており、同じ「選択」「追加済み」「候補」の状態表現が統一されていません。
3枚目の設定画面は、実装・デバッグ・ユーザー設定が混ざっています。Runtime / PWA の build 表示、notice、更新反映、一覧幅、A4 Hz、BPM、丸め、入出力が同じ設定画面にありますが、一般ユーザー視点では「音楽設定」「表示設定」「ファイル管理」「開発情報」が別カテゴリです。今のままだと、PWAやruntime情報が最上部にあるため、ユーザーはアプリの通常設定より先に内部状態を見せられる。これは開発中ツールとしては合理的ですが、完成UIとしてはノイズです。
ボタンの意味も揃っていません。「追加」「更新」「一括削除」「進行保存」「進行読込」「後ろに挿入」「Section追加」「チャンク追加」が並んでいますが、操作対象の粒度が違います。音高を追加するボタン、コードを進行へ追加するボタン、チャンクを追加するボタン、セクションを追加するボタンは、それぞれ対象階層が違うため、ラベルだけでは危険です。特に「追加」は何に追加するのかが文脈依存になりやすく、誤操作の温床になります。
UIとしての一番の問題は、内部データ構造がそのまま画面に出ていることです。P1、M3、P5、b7、M2、ratio、JI、degree、root、oct、chunk、section、activeNotes、project.mcb、library.mcbl など、仕様を知っている人には意味が通りますが、操作画面では「いま何を作っているのか」を中心に見せる必要があります。現状は「データモデル確認用UI」に近く、音楽的な作業フローへの翻訳がまだ弱いです。
さらに、画面密度の割に折りたたみ・優先表示・検索の使い分けが弱いです。音高プリセットやコード品質ボタンが大量に表示されるのは機能的には便利ですが、常時表示すると認知負荷が高くなります。特にコード品質ボタン群は、頻出コード、拡張コード、特殊コード、テスト用コードが同じ面に並んでいるため、初心者にも上級者にも少し扱いづらい。頻出順、タグ別、最近使用、検索、カテゴリ折りたたみなどがないと、ボタンの多さがそのまま迷いになります。
色設計も惜しいです。背景は綺麗ですが、機能的な色と装飾的な色が競合しています。ネオン系のアクセント、淡いボタン色、選択中の黄緑、カード背景、タグ色が混在しており、「色が意味を持っている」のか「ただの見た目」なのか判別しづらい。音楽ツールでは、色はルート、度数、コード品質、選択状態、警告、保存状態などに割り当てると強いですが、現状はその対応規則がまだ見えません。
要するに、今のUIは「機能が足りない」のではなく、むしろ逆で、機能・状態・内部概念が露出しすぎています。作曲支援ツールとしては、ユーザーが直接考えるべき層と、システム側が保持すべき層を分ける必要があります。画面上では「今のコード」「次に足す候補」「進行全体」「保存済み素材」だけを主役にし、ID、タグ、比率、cent、DSL由来メモ、PWA状態などは必要時に展開する設計の方が安定します。

【補足】
かなり簡単に言うと、現状は「全部入りの管理画面」になっています。作曲中に欲しいのは、まず「今どのコードを作っているか」「それをどこへ入れるか」「進行がどう流れているか」です。しかし今は、音高ID、タグ、メモ、プリセット、ライブラリ、保存、読込、PWA情報まで同じ視界に入るため、脳内で毎回フィルタリングが必要になります。UIがガバいというより、「内部仕様を確認する画面」と「実際に曲を組む画面」がまだ分離されていない、という見方が近いです。

【索引】
PWA: Webアプリを端末にインストール風に扱える仕組み。
activeNotes: 現在保持・再生対象になっている音高集合。
cent: 平均律上の半音を100等分した音程単位。
JI: Just Intonationの略で、整数比に基づく純正律体系。
DSL: 特定領域向けに設計された専用言語。
chunk: 進行データ内のまとまり単位として扱う区画。
section: 楽曲構造上の区分を表すまとまり単位。
runtime: アプリが現在動作している実行環境情報。
