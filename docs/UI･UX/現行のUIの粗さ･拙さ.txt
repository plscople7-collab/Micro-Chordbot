【概要】

* 現行UIは「横並びの成立条件」が揃っておらず、要素が並んでいるだけで、列・行・余白・基準線の統一が弱い。
* CSS的には、Grid/Flexの設計単位が画面全体・パネル・フォーム・リストで分断され、同じ種類の要素でも幅や高さが揃っていない。
* 入力欄、ボタン、ラベル、テーブル、カードのベース寸法が混在し、視線のガイド線が壊れている。
* 余白が「区切り」ではなく「空いてしまった隙間」に見える箇所が多く、画面密度に対して整列感が不足している。
* UIの粗さはデザイン装飾よりも、レイアウトの骨格、寸法規則、整列ルール、レスポンシブ方針の未統一に起因している。

【詳細解説】
現行UIの横並びで最も目立つ問題は、同じ行に置かれている要素同士の「論理的な列幅」が揃っていないことです。たとえば上部のオクターブ、スナップcent、cent、波形、音量などは横一列に配置されていますが、各フォームの幅・ラベル位置・入力欄の高さ・ボタンサイズが個別最適になっており、横方向のリズムが作れていません。見た目としては横並びですが、CSS構造としては「フォーム部品を順番に置いた」状態に近く、グリッドとしての列設計が弱い。特に、ラベル幅と入力欄幅の関係が揃っていないため、ユーザーの目線が左から右へ流れたときに、次に読むべき基準点が毎回ズレます。
2枚目のコード進行画面では、左の進行エディタと右の再生・進行表示領域が大きく2カラムになっていますが、この2カラムの意味づけがやや曖昧です。左は編集操作、右は再生・保存・進行カードという構造ですが、右上に操作群、右下にカード一覧があり、その間に大きな空白が発生しています。この空白は「余白として設計された抜け」ではなく、「コンテンツ量の違いで余っている領域」に見えるため、画面全体の締まりが弱くなります。CSS的には、右カラム内の縦方向レイアウトに `justify-content` 的な意図が見えず、上詰めの操作パネルと下方のカード群が分離してしまっています。
また、左カラムの内部でも横並びの粒度が揃っていません。Section、ルート、拍、臨時記号、Bass、転回形、度数ボタン、ルートボタン、コード品質ボタンがそれぞれ異なる幅規則で並んでおり、「フォーム行」「選択ボタン群」「タグ的ボタン群」が同じ面に混在しています。特にルートボタン列は横幅いっぱいに綺麗に見せようとしている一方、コード品質ボタン群は小さな矩形ボタンが詰め込まれており、同じ「選択肢」なのに表示密度が違います。これはCSSでいえば、ある領域は `grid-template-columns: repeat(...)` 的な均等配置、別領域は `flex-wrap` 的な詰め込み配置になっている印象で、選択UIとしての統一感が崩れています。
1枚目の音高ライブラリ周辺では、表の列幅が機能ではなく残り幅で決まっているように見えます。名前、cent、短名、タグ、メモ、操作ボタンが横に並んでいますが、タグ列とメモ列がかなり広く、操作ボタン列が窮屈です。その結果、重要度の高い「現在の音高」「cent」「短名」より、タグやメモの入力欄が視覚的に強くなっています。CSS的には、テーブル列に `minmax` や固定幅・可変幅の優先順位が設定されていない、または設定が実態に合っていない状態に見えます。UIとしては、識別列は短く、編集列は必要十分、メモ列は展開式にした方が安定します。
横並びの粗さは、基準線の不一致にも出ています。ラベルの上端、入力欄の上端、ボタンの中央線、タグの高さ、カードのタイトル位置が微妙に揃っていないため、画面全体が少し沈んだり浮いたりして見えます。これは色やフォント以前の問題で、`align-items: center`、`baseline`、`start` の使い分けが領域ごとにバラついている時に起きやすいです。フォーム行ではラベルと入力欄の関係を縦積みにするのか、横並びにするのかを統一し、ボタン行ではボタンの高さとアイコンの中心を揃える必要があります。
さらに、角丸・枠線・背景色のレイヤーが多く、横並びの境界が曖昧になっています。パネル、入力欄、ボタン、カード、タグがすべて角丸付きの暗色面を持っているため、どれが親コンテナで、どれが子要素で、どれが操作可能要素なのかが分かりづらい。CSS的には、コンポーネントごとに背景・枠線・角丸を持たせすぎており、階層の差が視覚的に薄い状態です。親パネルは薄い境界、入力欄は明確な内側面、ボタンは操作色、選択中はアクセント色、という具合にレイヤートークンを分ける必要があります。
設定画面では、横並びフォームの粗さがさらに分かりやすいです。「一覧幅」の各入力欄は横一列で綺麗に並んでいるように見えますが、各項目の幅がほぼ均等で、内容量に対する重みづけがありません。名前列rem、cent列rem、短名列rem、タグ列rem、メモ列rem、コード箱pxは、実際にはデータ種別も単位も違います。しかし同じ横並びにすることで、単位の違いが視覚上は同列扱いになります。ここは「テーブル列幅設定」という一まとまりの詳細設定であり、常時横一列に広げるより、列名・単位・数値・プレビューをセット化した方がCSS的にもUI的にも破綻しにくいです。
もう一つ大きいのは、レスポンシブ前提の曖昧さです。現行画面はPC幅では横に広がりますが、広がった分を意味ある領域に変換できていません。横幅があるから横に並べる、という配置になっており、横幅が増えた時に「何を広げるべきか」「何を固定すべきか」が明確でない。良いCSS設計では、ナビ・操作ボタン・数値入力などは固定幅寄り、リスト・プレビュー・進行キャンバスなどは可変幅寄りにします。現行UIでは、入力欄やメモ欄まで横に伸びすぎる一方、進行カードの配置領域は広い空白を作っており、可変領域の割り当てが逆転している箇所があります。
ボタン配置も、横並びのルールがまだ弱いです。「チャンク追加」「Section追加」「後ろに挿入」「削除」は同じ行にありますが、危険度・頻度・対象階層が違います。削除は破壊的操作なので、同じボタン群の右端に置くだけではなく、色・余白・区切りで分けるべきです。CSS的には同じ `.button` クラスで並んでいるような印象ですが、実際には primary、secondary、danger、utility の区別が必要です。横並びにするときほど、同じ行に置く操作は同じ対象・同じ頻度・同じ危険度に揃える必要があります。
現行UIの「ガバさ」は、個々の部品が雑というより、CSSの設計思想がページ単位で統一されていないことにあります。音高画面は表中心、コード画面はボタン群＋カード中心、設定画面はフォーム中心ですが、それぞれが別々のCSS規則で成り立っているように見えるため、同じアプリなのに画面ごとの密度・余白・列幅・操作感が変わります。ここを直すには、まず全画面共通で、`page-shell`、`toolbar`、`panel`、`form-grid`、`button-grid`、`data-table`、`card-grid`、`inspector` のようなレイアウト部品を定義し、それぞれの責務に合わせて幅・余白・整列・折返しを固定する必要があります。
理想的には、PC版は「2カラム＋下部ライブラリ」か「3ペイン構造」に寄せるべきです。左ペインは音高、中央ペインはコード作成、右ペインは進行・再生、下部またはサイドドロワーにプリセット。これなら横並びが単なる横配置ではなく、作業階層の可視化になります。現行のようにパネル内で細かく横並びを増やすより、まず大きなペインの意味を決め、その中でフォーム・ボタン・リストを規則的に配置する方がCSSも安定します。

【補足】
かなり実装寄りに言うと、現行UIは `flex` で都度並べた感が強く、画面全体の `grid` 設計が弱いです。Flexは「中身の自然な並び」には強いですが、音楽制作ツールのように列・行・階層・編集対象を明確にしたいUIでは、まず外枠をCSS Gridで切り、内側の小部品だけFlexにする方が安定します。現状は、横に並べられるものを横に置いた結果、同じ横並びでも「フォーム行」「操作列」「カード列」「設定列」の意味差が見えなくなっている状態です。

【索引】
CSS: Webページの見た目、配置、余白、色、寸法を定義するスタイル言語。
Flex: 要素を一方向に並べ、折返しや整列を制御するCSSレイアウト方式。
Grid: 行列構造で画面領域を分割するCSSレイアウト方式。
レスポンシブ: 画面幅や端末に応じて表示構造を変える設計方式。
ペイン: 画面内で特定機能を担う分割領域。
コンポーネント: UIを構成する再利用可能な部品単位。
