【概要】

* 現在UIは、機能量・実装量ではモックを大きく超えているが、UI/UXとしては「制作体験の一貫性」が後退している。
* 当初モックは「音高→コード→進行」を1つの流れとして扱う設計だったが、現状は「各機能の管理画面」の集合体に近い。
* 特にPC版で、本来モックが重視していた「一覧性」「空間記憶」「同時参照」が崩れている。
* 一方で、Undo/Redo、チャンク、保存、DSL対応、タグ、cent編集など、内部機能はかなり強化されており、構造自体は進化している。
* 現状は「高機能な内部エディタ」であり、モックが想定していた「音楽構築UI」へ翻訳し切れていない段階に見える。

【詳細解説】
まず、モック時点のUI思想はかなり明確です。PDFでは「全体を一覧出来るUI」と明示されており、PC版では左に音高系、右にコード系を配置し、同時に見ながら編集する構想でした。これは単なる見た目ではなく、「音高がコードを構成し、コードが進行を構成する」という内部構造を、そのまま空間配置へ変換する思想です。つまりモックの時点では、UIがデータ構造の可視化装置として機能していました。
現行UIは、この思想から少しズレています。現在は「音高」「コード進行」「設定」がタブで強く分離され、さらに各タブ内部でも、ライブラリ、編集、保存、設定、デバッグが混在しています。その結果、モックで成立していた「今作っている音が、どのコードになり、どの進行へ入るか」が空間的に追いづらくなっています。
特にUX的に大きいのは、「視線移動の意味」が変わった点です。モックでは、左下の音高プリセット→右下のコードプリセット→右上の進行という流れが、ほぼそのまま制作順になっています。ユーザーは「画面のどこを見れば次工程か」を自然に覚えられる。一方現行UIでは、音高を編集した後、コード進行タブへ移動し、別の編集系を開き、さらに進行へ積む必要があるため、制作フローが「画面遷移」に変換されています。これは一般的なCRUDアプリには向きますが、作曲ツールでは没入感を削りやすい。
また、モックは「空間記憶」をかなり重視しています。左は音高、右はコード、上は即時操作、下はライブラリ。この構造は、一度慣れると身体で覚えられる。一方現行UIでは、同じ概念が別タブ・別リスト・別フォームへ分散しているため、「あの機能どこだっけ」が発生しやすい。特にコード進行タブは、左にコード生成盤、右に進行表示、上に再生系、さらに保存読込が右上にあるため、情報の主従がやや曖昧です。
ただし、現行UIの方が優れている部分もかなりあります。
モック段階は思想図なので、「理想の流れ」はありますが、内部状態管理や大量データ運用への対応は弱い。一方現行UIは、

* activeNotes管理
* cent直接編集
* DSLタグ
* プリセット保存
* chunk
* Section
* Undo/Redo
* 波形選択
* スナップ
* round mode
* runtime/PWA
  など、実際の編集・保存・運用に必要な機能がかなり増えています。
  つまり、モックは「作曲体験の骨格」、現行UIは「実装機能の肉付け」に寄って進化しています。
  問題は、その肉付けが骨格を覆い隠し始めていることです。
  現在UIは、「何ができるか」は強くなった一方、「何を作っているか」が少し見えづらくなっています。特に音楽系ツールでは、内部状態を全部見せると途端に“DAWの設定画面感”が出ます。μChordbotの本来の強みは、「複雑な理論を持ちながら、触感としてはChordbotのように軽い」点だったはずですが、現状は理論層がかなり露出しています。
  モックと比較すると、UI密度の質も変わっています。モックは手書きながら、情報が「島」で分かれている。各領域が大きな意味単位で区切られているため、多少雑でも理解しやすい。一方現行UIは、細かいボタン、入力欄、タグ、表、カードが増えたことで、画面密度が「編集項目密度」になっています。つまり情報量は増えたが、認知的には圧縮されていない。
  さらに、モックでは「コード進行」が画面の主役です。右上のボックス群がかなり大きく描かれており、コードを積んでいく感覚が中心にあります。しかし現行UIでは、進行カード群よりも、左側のコード構築パラメータ群の方が存在感が強い。結果としてUXが「曲を並べる」より「コード定義を編集する」方向へ寄っています。
  CSS/レイアウト面でも、モックは「大枠が少ない代わりに意味が明快」です。現行UIはパネル分割・入力欄・カード・テーブルが増えたため、境界線が増え、視覚ノイズも増えています。特に、同じ濃紺系UI内で、入力欄、パネル、カード、テーブルが全部似た色調を持っているため、画面全体が“沈む”印象があります。モックの方は白背景＋大きな領域区切りなので、構造が直感的に追いやすい。
  総合すると、現在UIは「機能の進化」に成功しているが、「制作体験の統合」が追いついていない状態です。モックの時点では、“音楽を組み立てる机”だったものが、現状では“高機能な音楽データ編集環境”に寄っています。これは悪化ではなく進化途中ですが、当初のモックが持っていた「全体を一覧しながら、触って組み立てる」という思想は、かなり重要だったと言えます。

【補足】
かなり端的に言うと、
モック:
「音楽を組み立ててる感覚」
現行:
「音楽データを編集してる感覚」
です。
しかも現行は、内部構造としてはかなり強いので、逆にUI層だけ整理されると一気に化ける段階まで来ています。内部機能は既に“楽器”寄りなのに、見た目と操作感がまだ“管理画面”寄り、というズレがあります。

【索引】
CRUD: 作成・読取・更新・削除を中心とする管理画面型設計。
UX: 操作時の認知負荷や体験全体を含む利用感覚設計。
Chordbot: コード進行を直感的に積み上げる作曲支援アプリ。
activeNotes: 現在保持・再生対象として扱われる音高集合。
DSL: 特定用途向けに設計された専用言語体系。
Undo/Redo: 編集履歴を戻す・やり直す操作機構。
chunk: 進行データを分割管理するまとまり単位。
Section: 曲構造上の区間単位。Aメロやサビ等を含む。
