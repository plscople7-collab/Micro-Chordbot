# UI / UX ポリシー

## 設計原則
- データベースのテーブルや内部変数名など実装の複雑さをユーザーに見せず、UIはユーザーの意図やドメイン概念を反映する。
- 内容を減らすのではなく、ユーザーが覚えたり比較したり検索したり保持したりする負担を減らす。
- 関係や階層を構造で示し、テーブルやフォームを乱用せず、可能な限り直接操作型のUIにする。
- UIは情報閲覧だけでなく創作のための道具であり、CRUD管理画面にしない。
- 編集や操作時には即時の視覚フィードバックを行い、外部トリガーが必要な場合は明示的に配置する。
- 編集中の対象を明確にし、選択中の項目を強調し、操作部の位置を一貫させる。
- 操作を可逆にし、Undo/Redoを実装し、ユーザーが手動でバックアップする必要をなくす。
- 参照切れや無効状態のデータは自動削除せず、問題を視覚的に示し、修復手段を提供する。
- PC用とモバイル用のUIを別々に設計し、単純に縮小しない。
- レイアウトやタイポグラフィ、色についてデザイントークンを用い、一貫性を保つ。
- 状態変化や作用範囲、関係を説明文に頼らず、視覚的に把握できるようにする。
- 多方向スクロールやネストされたスクロール領域、隠れた状態を避け、できる限り単方向スクロールで扱えるようにする。
- タグは分類用途にのみ用い、エンティティの属性そのものをタグへ押し込まない。
- 編集・閲覧・デフォルトなど複数の状態を簡単に切り替えられるようにし、別画面への遷移を必要最小限にする。

## レイアウトとインタラクション
- テーブルは万能ではないことを認識し、複雑な関係や階層は適切な構造で表現する。
- システムが把握している情報をユーザーに覚えさせたり再入力させたりしない。
- データ変化時には明示的な更新ボタンを要求せず、リアルタイムで更新する。

## 挙動
- 編集は別画面やフォームではなくインラインまたはコンテキスト内で行う。
- 変更は即座に反映され、Undo/Redoで取り消せるようにする。
- 無効入力時は処理を停止せず、自動補正やフォールバック処理を行う。
- 変更箇所を強調し、ユーザーの位置を維持し、隠れたスクロールバーや予期しない再配置を避ける。
- 汎用的なプレースホルダよりもドメインに合わせた適切なデフォルト値や候補を提示する。
- 一般的な記号やラベルなど、広く理解されているアイコンや表現を用いる。
